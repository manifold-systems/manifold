\title{Compositional Polymorphism:\\
Resolving the Object Composition Trilemma}
\author{R. Scott McKinney\\
\textit{scott@manifold.systems}}
\date{}
\maketitle

\begin{abstract}
We identify a fundamental trilemma in object-oriented polymorphism: self-call
correctness, independent late-bound components, and static typing. No prior model
satisfies all three simultaneously. We
present \emph{compositional polymorphism}, a composition model that satisfies
all three properties. The model is realized as a compiler plugin for Java, supports arbitrary DAG
composition with per-interface dispatch roots, introduces composite-scoped
encapsulation, and delivers dispatch performance equivalent to standard virtual
dispatch.
\end{abstract}

%% ─────────────────────────────────────────────────────────────────────────────
\section{Introduction}
%% ─────────────────────────────────────────────────────────────────────────────

Object-oriented programming promises polymorphic behavioral reuse. Every
mechanism it has produced for delivering that promise sacrifices at least
one of three properties:

\begin{enumerate}
  \item \textbf{Self-call correctness} --- in all compositions, self-calls within
        a component dispatch through the composite host (\emph{interface-scoped
        dispatch}; the \emph{Self problem}~\cite{lieberman1986,self1987}: a
        self-call in a component should invoke the composite's override, not the
        component's own implementation)
  \item \textbf{Independent, late-bound components} --- components are
        independent objects supplied at construction time, not statically inlined,
        fused, or ordered into a linear chain; a component may simultaneously
        carry a distinct composite identity per interface
  \item \textbf{Static typing} --- the composition's interface contracts are
        enforced at compile time
\end{enumerate}

Prototype-based delegation~\cite{lieberman1986,self1987} solves the Self problem
and provides late-bound components, but without static type guarantees. Scala
traits~\cite{odersky2005} and Rust trait impls enforce static typing and support
polymorphism, but collapse to a single linearized type at link time---real
components are gone. Python's MRO and Ruby's mixins linearize the chain, forfeiting independent
component identity and, in the case of dynamic languages, static typing. Java
interfaces with default methods provide static typing but no delegation
semantics. Kotlin's \code{by} keyword provides forwarding, not delegation---self-calls
do not dispatch through the composite.

Even the closest prior models fall short.
Kniesel's type-safe delegation~\cite{kniesel1999} addresses the Self problem
under static typing but restricts each component to a single composite identity,
failing P2.
Ostermann's delegation layers~\cite{ostermann2002} achieve correct self-call
dispatch and static typing, but components must be defined as virtual nested
classes of a common enclosing family type---independently defined classes
cannot be composed.

These failures are not academic: they are why implementation inheritance
remains in widespread use despite decades of criticism~\cite{gamma1994}.
When composition cannot preserve self-call correctness, practitioners
reach for inheritance because it does. Compositional polymorphism closes
that gap.

This paper presents \textbf{compositional polymorphism}, which satisfies all
three properties --- and does so at dispatch cost equivalent to standard virtual
dispatch, making the model practical for production use. The
core contributions are:

\begin{itemize}
  \item \textbf{The Object Composition Trilemma} --- we identify and formalize
        the trilemma (self-call correctness, independent late-bound components,
        static typing) and establish that no prior model satisfies all three
        simultaneously.
  \item \textbf{Interface-scoped dispatch} --- we introduce this dispatch
        primitive, in which the composition receiver is determined per interface
        rather than per component, and show it is the structural change that
        resolves the trilemma.
  \item \textbf{A formal model} --- compositional polymorphism is defined over
        a standard typed object model with a central dispatch invariant
        (Proposition~1) and composite-scoped encapsulation (\code{@internal}).
  \item \textbf{A practical implementation} --- a \texttt{javac} plugin for
        Java 8--26 realizing the model at O(1) dispatch cost, with open-source
        release and full IDE integration.
\end{itemize}

\section{Overview}
%% ─────────────────────────────────────────────────────────────────────────────

Compositional polymorphism introduces two annotations. \code{@part} marks a
class as a reusable component; \code{@link} marks a field through which one
or more interfaces are delegated. A \code{@part} class implements one or more
interfaces; when wired into a composite via a \code{@link} field, its
self-calls for those interfaces dispatch through the composite host rather
than through the component itself.

Consider a minimal example:

\begin{lstlisting}
interface Formatter {
    String prefix();
    String format(String msg);
}

@part class LogFormatter implements Formatter {
    public String prefix() { return "INFO"; }
    public String format(String msg) { return "[" + prefix() + "] " + msg; }
}

class AppFormatter implements Formatter {
    @link Formatter fmt;
    AppFormatter(Formatter f) { this.fmt = f; }
    @Override public String prefix() { return "APP"; }
}

new AppFormatter(new LogFormatter()).format("started"); // "[APP] started"
\end{lstlisting}

When \code{AppFormatter} is constructed, the compiler wires
\code{LogFormatter}'s composite identity for \code{Formatter} to
\code{AppFormatter}. A call to \code{format("started")} dispatches to
\code{LogFormatter.format()}, and the self-call \code{prefix()} within
\code{format()} dispatches back through \code{AppFormatter}---invoking
\code{AppFormatter.prefix()}, not \code{LogFormatter.prefix()}.
This is self-call correctness: the composite's override wins, not the
component's own implementation.

Because the \code{@link} field accepts any \code{Formatter} at construction
time, the component is a real runtime object---not fused into the class
hierarchy. A different \code{Formatter} implementation can be substituted
without changing \code{AppFormatter}. This is independent late-bound
components.

All of this is enforced at compile time. \code{@link} fields are
\code{private final}; the compiler rejects any use of \code{this} in a
\code{@part} class that would allow the component's identity to escape the
composite dispatch path. This is static typing.

The formal model in the following section generalizes these ideas to DAG
composition, where a single component carries distinct composite identities
for multiple interfaces simultaneously.

\section{The Model}

\subsection*{Motivating example}

In the model, \code{@part} annotates component classes and \code{@link}
annotates delegation fields; both are defined formally in the following
section. The following example requires all three trilemma properties to
produce the correct output and exposes the limits of every prior model.
\code{WorkerPart}
implements both \code{Logger} and \code{Processor} and, when composed into
\code{App} via \code{LoggingWorker}, carries distinct composite identities for
each interface---simultaneously rooted at \code{LoggingWorker} for
\code{Logger} dispatch and at \code{App} for \code{Processor} dispatch
(Figures~\ref{fig:dag-diagram} and~\ref{fig:dag-listing}).

This example is not expressible with any prior model in Table~\ref{tab:comparison}.
With Kotlin's \code{by}, self-calls in \code{WorkerPart} resolve within
\code{WorkerPart} itself --- \code{prefix()} returns \code{"WORKER"} and
\code{label()} returns \code{"worker"} regardless of the composite host, because
\code{by} is forwarding, not delegation. With Scala traits, the example
exposes a structural dilemma. To accept a runtime \code{Processor} as a
constructor argument, the Scala programmer must use a concrete object and
forward manually---the same forwarding hazard as Kotlin \code{by}, with the
same result: self-calls in \code{WorkerPart} resolve within \code{WorkerPart}
and \code{label()} returns \code{"worker"}, not \code{"APP"} (P1 fails). The
only alternative is to fuse \code{WorkerPart} via trait inheritance---but then
no runtime object can be passed as a parameter at all (P2 fails). There is no
Scala arrangement that satisfies both. All three trilemma properties are required to produce the
observed output. The following definitions formalize the model.

\subsection*{Definitions}

We define compositional polymorphism over a standard typed object model where
classes implement interfaces and method dispatch is defined by the receiver's
runtime type.

\begin{definition}[Component]
A \emph{component} is a class annotated \code{@part}. Components implement one
or more interfaces. A component may contain \code{@link} fields (making it a composite).
\end{definition}

\begin{definition}[Link]
A \emph{link} is a field annotated \code{@link} in a host class. A link field
is \code{private} and \code{final}. The \emph{delegation surface} of a link is
the intersection of the field's declared type and the interfaces of the
enclosing class (or the subset specified by \code{value()}).
\end{definition}

\begin{definition}[Composite]
A \emph{composite} is a class containing one or more \code{@link} fields. A
composite implements the union of the delegation surfaces of all its links.
\end{definition}

\begin{definition}[Delegation surface]
The delegation surface of a composite is the set of interfaces delegated by at
least one link. Each interface in the surface is assigned to exactly one link;
ambiguity (diamond overlap) is a compile error unless resolved with
\code{share=}.
\end{definition}

\begin{definition}[Composite identity]
When a component is wired to a composite, it receives a \emph{composite
identity}: a reference to the composite object. Self-calls within the
component---calls to methods in the component's interfaces---dispatch through
the composite identity rather than \code{this}.
\end{definition}

\begin{definition}[Interface-scoped dispatch]
\label{def:isd}
\emph{Interface-scoped dispatch} is the dispatch mechanism by which the
composition receiver for a self-call is determined by the interface through
which the call is routed, not by the object identity of the caller. Each
interface in a component's delegation surface is assigned an independent
composite identity; self-calls route through the composite host for that
interface.
\end{definition}

\begin{definition}[Transitive composability]
A composite may itself be a component. When a composite-component is wired to a
larger composite, composite identity propagates transitively: each interface in
the composite-component is re-rooted at the new outermost composite.
\end{definition}

\begin{definition}[Delegation surface restriction]
An interface in a component's \code{implements} clause may be annotated
\code{@internal} to exclude it from the component's delegation surface. An
\code{@internal} interface participates in the component's own behavior but
cannot be delegated by external composites.
\end{definition}

\begin{definition}[Composite-scoped method]
An interface method annotated \code{@internal} is accessible within the
composite family (component-to-component via link fields) but is not accessible
through the composite's external interface. It is the compositional analog of
\code{protected} without package-scoping.
\end{definition}

\begin{definition}[Abstract component]
A component may be declared \code{abstract}. Abstract methods in the component
are obligations imposed on the delegating composite. The compiler generates a
static \code{asLink()} factory per non-private constructor; \code{asLink()}
returns a \emph{deferred} instance whose abstract methods throw
\code{DelegationLinkageError} until wired.
\end{definition}

\begin{definition}[Diamond resolution]
\label{def:diamond}
When two links in a composite share a common interface, exactly one must declare
\code{share=T} for the shared interface. That link is authoritative; the other
delegates only its non-shared interfaces. Unresolved diamonds are compile
errors.
\end{definition}

\begin{proposition}
For any component $P$ in a delegation chain for interface $I$ rooted at
composite $C$: a self-call \code{this.m()} within $P$ where $m \in I$
dispatches to $C$.
A formal proof in a small-step operational semantics is left to future work
(Section~\ref{sec:futurework}).
\end{proposition}

The guarantee holds because wiring is completed before the composite is
observable: at any point a self-call can occur, the dispatch slot for $I$
already holds the outermost composite in the chain, so dispatch is correct
by construction.

%% ─────────────────────────────────────────────────────────────────────────────
\section{Implementation}
%% ─────────────────────────────────────────────────────────────────────────────

The implementation is a \texttt{javac} plugin operating on the compiler's AST
during two phases. In \textsc{enter}, the plugin generates structural
scaffolding: the \code{\$selves} composite identity array and
\code{\$linkPartToSelf} wiring method for each \code{@part} class, delegation
method skeletons for each composite class, and (for abstract components) the
\code{\$Impl} subclass and \code{asLink()} factories. In \textsc{analyze},
after type resolution, self-calls in \code{@part} bodies are rewritten,
\code{@link} field assignments are rewritten to invoke wiring, and
\code{@internal} visibility is enforced. No bytecode manipulation is performed;
all transformation is at the AST level and subject to standard type checking.

\subsection{Component Transformation}

Each \code{@part} class holds a \code{\$selves} array indexed by interface
ordinal, assigned at compile time:

\begin{lstlisting}[language={},style=java]
$selves: Object[]
// initialized: $selves[IDX_I] = this  (before wiring)
// after wiring: $selves[IDX_I] = composite identity for interface I
\end{lstlisting}

An unwired component dispatches on itself; once wired, dispatch routes through
the composite identity. \code{\$linkPartToSelf} updates each relevant slot and
recurses through any \code{@link} fields in the component, propagating identity
transitively through the DAG. Cycle detection is performed at each slot
assignment.

\subsection{Rewriting}

Within a \code{@part} method body, every call \code{this.m(args)} where
\code{m} is declared in interface $I$ is rewritten to:

\begin{lstlisting}
((I) $selves[IDX_I]).m(args)
\end{lstlisting}

In composite classes, each \code{@link} field assignment \code{this.link = expr}
is rewritten to invoke \code{\$linkPartToSelf}, propagating the composite
identity into the component. An O(1) marker interface check guards the call so
that non-\code{@part} assignments pass through unchanged.

\paragraph{Self-preservation.}
A \code{@part} class may not reference \code{this} except as one of its
declared interfaces; passing or assigning \code{this} as the concrete type are
compile errors. Field access via \code{this} is permitted---a part may call
back through its own \code{@link} fields, the compositional analog of
\code{super}. If \code{this} leaked as the concrete type, a caller could invoke
methods outside the composite dispatch path, violating Proposition~1.

\subsection{Abstract Component Support}

For an abstract \code{@part} class \code{Foo}, the compiler generates a
private \code{\$Impl} subclass whose abstract method stubs throw
\code{DelegationLinkageError}, and an \code{asLink()} factory per non-private
constructor:

\begin{lstlisting}
private static final class $Impl extends Foo {
    @Override public T m(Params p) {
        throw new DelegationLinkageError("abstract method called before linkage");
    }
}
public static Foo asLink(...) { return new $Impl(...); }
\end{lstlisting}

Once wired, the stubs are unreachable: self-calls dispatch through
\code{\$selves[IDX\_I]}, which resolves to the composite.

\subsection{\code{@internal} Enforcement}

\code{@internal} interfaces are excluded from the component's delegation
surface; composite classes cannot expose them via \code{@link} fields.
\code{@internal} methods are excluded from the composite's external interface;
calls from outside the composite family are compile errors, while calls via
\code{@link} field references within the family are permitted. All enforcement
is at compile time with no runtime overhead.

\section{Evaluation}
%% ─────────────────────────────────────────────────────────────────────────────

\subsection{Dispatch Performance}

The dispatch path for a self-call through a composite is:
\begin{enumerate}
  \item Load \code{\$selves[IDX\_I]} --- array indexed load, compile-time
        constant index
  \item Cast to $I$
  \item \code{invokeinterface m} --- standard JVM virtual dispatch
\end{enumerate}

This is functionally O(1). The parallel to vtable dispatch is direct: a vtable
call is an indexed slot load followed by an indirect call. In practice, JIT
compilers inline trivial forwarders entirely. Composite DAGs are shallow and their call sites are monomorphic or bimorphic
in the common case---the JIT's most aggressively inlined scenarios. Composite
dispatch performance is therefore equivalent to direct virtual dispatch in
practice.

\subsection{Expressiveness}

The model supports four composition patterns not achievable together in prior
static OO models:

\begin{enumerate}
  \item \textbf{Independent component identity} --- components are real objects;
        their internal state is independent
  \item \textbf{Interface-scoped dispatch} (Definition~\ref{def:isd}) --- self-calls
        route through the outermost composite in the delegation chain for the
        dispatched interface
  \item \textbf{Multiple composite roots per component} --- a component wired
        for multiple interfaces simultaneously carries distinct composite
        identities per interface
  \item \textbf{Composite-scoped encapsulation} --- \code{@internal} provides
        finer-grained visibility than \code{public}/\code{protected}/\code{private}
        for component families
\end{enumerate}

\subsection{Case Study: Teaching Assistant}

The following scenario exercises DAG composition, diamond dispatch resolution,
and self-call correctness simultaneously. A teaching assistant is both a student
and a teacher; both roles share a common \code{Person} base.

\begin{lstlisting}
interface Person  { String getName(); String getTitle(); String getTitledName(); }
interface Student extends Person { String getMajor(); }
interface Teacher extends Person { String getDept();   }
interface TA      extends Student, Teacher {}
\end{lstlisting}

\paragraph{Kotlin \code{by}.}
Kotlin's built-in \code{by} clause satisfies P2---the implementation is
injected at construction time as a real object. P1 does not hold. Delegating
\code{Student by s} and \code{Teacher by t} creates a diamond: the compiler
forces a manual override of every shared \code{Person} method. More critically,
\code{getTitledName()}---the template method worth sharing---cannot be safely
delegated: whichever branch receives it calls \code{getTitle()} on its own
\code{this}, bypassing \code{TaImpl.getTitle()}.

\begin{lstlisting}[style=kotlin]
class TaImpl(val s: Student, val t: Teacher) : TA,
    Student by s, Teacher by t {
  // Diamond: every shared Person method must be manually resolved
  override fun getName()  = s.getName()
  override fun getTitle() = "TA"
  // Without this, s.getTitledName() calls s.getTitle() = "Student", not "TA"
  override fun getTitledName() = "${getTitle()} ${getName()}"
}
\end{lstlisting}

Every composite in the hierarchy must re-implement \code{getTitledName()}.
Adding a new template method to \code{Person} requires a matching override in
every composite class. \code{by} is forwarding with cleaner syntax; the
composition does not scale.

\paragraph{Compositional polymorphism.}
\code{PersonPart} holds the template once. Its self-call to \code{getTitle()}
dispatches through the composite host at each level of the delegation chain.

\begin{lstlisting}
@part class PersonPart implements Person {
  private final String name;
  public PersonPart(String name) { this.name = name; }
  public String getName()       { return name; }
  public String getTitle()      { return "Person"; }
  public String getTitledName() { return getTitle() + " " + getName(); }
}
@part class StudentPart implements Student {
  @link Person person;
  public StudentPart(Person p, String major) { ... }
  public String getTitle() { return "Student"; }
}
@part class TeacherPart implements Teacher {
  @link Person person;
  public TeacherPart(Person p, String dept) { ... }
  public String getTitle() { return "Teacher"; }
}
@part class TaPart implements TA {
  @link(share=Person.class) Student student;
  @link Teacher teacher;
  public TaPart(Student s) {
    this.student = s;
    this.teacher = new TeacherPart(s, "Math");
  }
  public String getTitle() { return "TA"; }
}

TA ta = new TaPart(new StudentPart(new PersonPart("Milton"), "CS"));
ta.getTitledName(); // "TA Milton"  -- getTitle() dispatches through TaPart
\end{lstlisting}

\code{getTitledName()} is defined once in \code{PersonPart} and overridden
nowhere. Diamond resolution is handled statically: \code{@link(share=Person.class)}
designates \code{student} as the authoritative delegate for \code{Person},
so \code{teacher} inherits the same \code{Person} identity rather than
introducing a second one. The composition graph is a proper DAG.

Kotlin \code{by} satisfies P2 but not P1: self-calls in a delegated class
dispatch through the delegated object, not the composite host. Compositional
polymorphism satisfies all three properties; interface-scoped dispatch
(Definition~\ref{def:isd}) is the mechanism that makes this possible.

\subsection{Test Coverage}

The implementation is tested against a suite covering:
\begin{itemize}
  \item forwarding vs.\ delegation semantics
  \item DAG composition at depth $\geq 3$
  \item diamond detection and resolution
  \item abstract component / \code{asLink()} lifecycle
  \item \code{@internal} interface and method enforcement
  \item default method dispatch
  \item inheritance chains in \code{@part} hierarchies
  \item cycle detection (runtime \code{DelegationLinkageError})
\end{itemize}

\subsection{Limitations}

\begin{itemize}
  \item Using \code{@link}, \code{@part}, and \code{@internal} requires
        configuring the \texttt{javac} plugin in the build system (Maven,
        Gradle, or similar).
  \item Composite dispatch semantics are not enforced under Java reflection.
        This is by design.
  \item The plugin relies on internal \texttt{javac} APIs for AST
        rewriting. These APIs are not part of the official public compiler API
        and may change across JDK versions. This is a practical constraint of
        the \texttt{javac} plugin approach; it does not affect the model's
        correctness or generality.
\end{itemize}

\subsection{Safety Guarantees}

\paragraph{Cycle detection.}
If a component is wired to itself directly or transitively,
\code{De\-le\-ga\-tion\-Link\-age\-Er\-ror} is thrown at assignment time during
the recursive \code{\$linkPartToSelf()} call.

\paragraph{Deferred method access.}
If an \code{asLink()}-produced instance has an abstract method called before
wiring, \code{DelegationLinkageError} is thrown at the stub. This is the
compositional analog of \code{AbstractMethodError}.

\paragraph{\code{@internal} violations.}
All \code{@internal} enforcement is at compile time; no runtime overhead.

\paragraph{Self-preservation.}
\code{@part} classes are not permitted to reference \code{this} in a context
other than a declared interface. This statically prevents component identity
from escaping the composite dispatch path, ensuring Proposition~1 holds for
all reachable call sites. Field access via \code{this} is permitted.

%% ─────────────────────────────────────────────────────────────────────────────
\section{Discussion}
%% ─────────────────────────────────────────────────────────────────────────────

\subsection{Structural Root Cause}

The design space reveals a progression, borne out by the survey in
Section~\ref{sec:related}. Prior models fall into three categories.
\emph{Single-receiver} models---mixins, traits, Kotlin \code{by}---treat
the implementation receiver and the composition receiver as the same object.
Mixins and traits satisfy P1 within the structures they support; Kotlin
\code{by} fails P1 outright, routing self-calls within the delegated object
rather than the composite host.
Multiple inheritance occupies a distinct position: vtable merging preserves
P1 (one object, virtual dispatch intact) but destroys independent component
identity, failing P2 through object fusion rather than dispatch restriction.
\emph{Delegation-based} models---Kniesel's type-safe delegation, delegation
layers, virtual classes---route self-calls through a composite host,
satisfying P1, but restrict each component to a single composite identity;
independently defined components cannot be composed, failing P2.
\emph{Interface-scoped dispatch} (Definition~\ref{def:isd})---compositional
polymorphism---assigns dispatch identity per interface rather than per object,
breaking the assumption that makes P2 intractable. This is the structural
change that makes all three properties simultaneously achievable.

\subsection{Language Design Implications}

The trilemma is not fundamental to statically-typed OO languages per se; it
is a consequence of how composition is currently specified and implemented in
those languages. Extending any statically-typed OO language
with a composition model equivalent to \code{@link}/\code{@part} would suffice.

The result has a direct implication for the long-standing inheritance debate:
implementation inheritance is no longer a necessary compromise. Compositional
polymorphism directly addresses each well-documented failure mode:

\begin{itemize}
  \item \textbf{Fragile base class} --- components are independent objects;
        changes to one do not silently affect others.
  \item \textbf{Forced IS-A semantics} --- reuse does not require a type
        relationship; the composite IS-A the interface, not the component.
  \item \textbf{The Self problem} --- a hazard introduced when composition
        replaces inheritance; resolved by routing self-calls through composite
        roots.
  \item \textbf{Diamond ambiguity} --- explicit \code{share=} resolution;
        unresolved diamonds are compile errors, not silent merges.
  \item \textbf{Encapsulation breach} --- \code{@internal} provides
        component-family visibility without exposing internals across
        package boundaries.
  \item \textbf{Abstract behavior} --- abstract components with deferred
        linkage (\code{asLink()}) preserve the template-method capability
        of abstract classes without coupling the hierarchy.
\end{itemize}

Interface inheritance, which provides type contracts and is foundational
to this model, is untouched. The self-preservation constraint---that a
\code{@part} class may not expose \code{this} as its concrete type---is
not a limitation but the static guarantee that makes the dispatch invariant
hold; implementation inheritance offers no equivalent assurance.

The Java specifics that make the plugin approach tractable:
\begin{itemize}
  \item \texttt{javac} plugins have full AST read/write access during
        compilation;
  \item \code{invokeinterface} in the JVM supports interface-typed dispatch
        independently of the concrete class hierarchy;
  \item interfaces as first-class types allow the \code{\$selves} array to be
        typed at the interface level.
\end{itemize}

\subsection{Concurrency}

The DAG structure is favorable for concurrent use. Component wiring occurs
during construction, before the object is shared. Once wired, \code{\$selves}
is effectively immutable---no synchronization is required for dispatch.
Components may contain mutable state, subject to the usual Java concurrency
rules; the dispatch mechanism adds no additional synchronization points.

\subsection{Future Work}
\label{sec:futurework}

\paragraph{Intersection type syntax.}
The natural expression of multi-interface linking---\code{@link Foo \& Bar
fooBar;}---would eliminate the \code{value()} workaround. This requires
syntactic extension to the Java parser within the \texttt{javac} plugin;
feasibility is under investigation.

\paragraph{Formal verification.}
A formal soundness proof for the dispatch model (Proposition 1) in a small-step
operational semantics would strengthen the theoretical foundation. The model
maps cleanly to a subset of Featherweight Java; this is planned as follow-on
work.

\paragraph{Role-oriented programming.}
The \code{@internal} mechanism partially overlaps with role-oriented programming
models. A formal comparison is left to future work.

\paragraph{Per-interface composite roots as a general primitive.}
The structural insight underlying this model---that dispatch identity can be
assigned per interface rather than per object---opens a solution space that
prior single-composite-identity models cannot reach. Problems currently addressed
suboptimally through dynamic dispatch, AOP weaving, or collapsed component
hierarchies may admit statically sound solutions within this primitive. This
is left as an open research direction.

%% ─────────────────────────────────────────────────────────────────────────────
\section{Related Work}
\label{sec:related}
%% ─────────────────────────────────────────────────────────────────────────────

\paragraph{Prototype-based delegation.}
Lieberman~\cite{lieberman1986} introduced the delegation concept: an object may
delegate unanswered messages to a prototype, with \emph{self} remaining the
original receiver. Self~\cite{self1987} realized this in a working language.
These systems solve the Self problem and support late-bound components but
provide no static type guarantees and rely on hash-based message dispatch,
failing P3.

\paragraph{Behavioral subtyping.}
Liskov and Wing~\cite{liskov1994} established the substitutability principle
underpinning interface contracts in statically-typed OO. This is a correctness
criterion, not a composition mechanism, and so does not appear in the trilemma
comparison.

\paragraph{Mixins.}
Bracha and Cook~\cite{bracha1990} formalized mixins as parameterized abstract
subclasses. Mixins extend through linearization---the mixin is applied to a
chain, not composed with independent objects. Self-calls dispatch through
\code{this} in the linearized result, satisfying P1 within the structures
mixins support. There are no independent runtime component objects, failing P2.

\paragraph{Traits.}
Sch\"arli et al.~\cite{scharli2003} proposed traits as stateless, composable
units of behavior. Trait composition is a compile-time operation: methods are
flattened into the target class. Self-calls dispatch through \code{this} in the
composed class, satisfying P1 within the structures traits support. There are
no independent runtime component objects, failing P2.

\paragraph{Scala traits / abstract type members.}
Odersky and Zenger~\cite{odersky2005} introduced statically-typed mixin
composition with abstract type members. Scala's \code{with} clause linearizes
traits into the target class at compile time. Late binding is limited to the
linearized type---independent components as runtime objects are not preserved.

In simple compositions where all of a component's interfaces are delegated
together, Scala traits satisfy P1 correctly: self-calls dispatch through
the outermost class in the linearized hierarchy. The limitation is structural
and emerges in the general case: a composite cannot hold its own implementation
of an interface while selectively delegating a different interface from the
same component, because delegation is all-or-nothing. Without selective
delegation, per-interface dispatch roots are inexpressible, and P1 fails in
that general case. P2 fails entirely---and this is not coincidental. The
linearization that enables P1 in the common case requires fusing the component
into the class hierarchy; that fusion is precisely what eliminates independent
late-bound component objects. In any linearization-based system, satisfying
P1 through linearization structurally precludes P2.

\paragraph{Multiple inheritance.}
Stroustrup~\cite{stroustrup1987} introduced virtual base classes and virtual
inheritance in C++ to handle diamonds. The result is a single linearized object
with a merged vtable. Independent component identity is lost, failing P2.

\paragraph{Kniesel / Lava.}
Kniesel's ``type-safe delegation''~\cite{kniesel1999} (ECOOP 1999) addresses
the Self problem under static typing using immutable delegation links in the
Lava language, satisfying P1 within single-rooted compositions and P3. Lava's
delegation model is single-rooted---a component cannot carry independent
composite identities for distinct interfaces simultaneously---failing P2.

\paragraph{Delegation Layers.}
Ostermann's delegation layers~\cite{ostermann2002} (ECOOP 2002) achieve
correct self-call dispatch and enforce type safety, satisfying P1 within
single-rooted compositions and P3---but only within a family constraint:
components must be defined as virtual nested classes of a common enclosing
owner type. Independently defined classes from unrelated hierarchies cannot
be composed, failing P2.

\paragraph{Virtual classes and family polymorphism.}
The virtual-class lineage---gbeta, CaesarJ, Object Teams/Java, and
J\&~\cite{nystrom2006}---achieves open class composition through virtual nested
classes within a common enclosing family type; Ernst, Ostermann, and
Cook~\cite{ernst2006} provide the definitive formal treatment.
Self-calls dispatch correctly within the family and static typing is enforced,
satisfying P1 within single-rooted compositions and P3. P2 fails across the entire lineage: components are
class-level members of an enclosing family type, not independent objects wired
at construction time. Introducing a new component combination requires defining
a new family subtype---it is class-level inheritance, not runtime object assembly.

\paragraph{Dynamic composition.}
Newspeak~\cite{bracha2010} treats classes as first-class late-bound arguments,
enabling runtime composition of independent module objects with correct
self-call dispatch, satisfying P1 and P2. Static type checking is absent by
design, failing P3. This is the strongest known example of a language that
deliberately satisfies P1 and P2 at the cost of P3.

\paragraph{Kotlin \code{by}-delegation.}
Kotlin's \code{by} keyword~\cite{kotlin} generates forwarding methods at compile
time. The delegated object's self-calls resolve within the delegated
object---the composite host is invisible. This is forwarding, not delegation, failing P1.

%% ─────────────────────────────────────────────────────────────────────────────
\section{Conclusion}
%% ─────────────────────────────────────────────────────────────────────────────

We have presented compositional polymorphism, a model that satisfies all three
properties of the trilemma --- self-call correctness, independent late-bound
components, and static typing --- realized as a compiler plugin for Java. It
operates at compile time via AST rewriting, delivers dispatch performance
equivalent to standard virtual dispatch, and supports DAG composition to
arbitrary depth, composite-scoped encapsulation, and abstract components with
deferred linkage.
The key structural contribution is \emph{interface-scoped dispatch}: assigning
the composition receiver per interface rather than per component, replacing the single composite identity assumption of prior models. The trilemma is not
fundamental to statically-typed OO languages; it is a consequence of that
assumption. Any statically-typed OO language extended with an equivalent
composition model would satisfy all three
properties.
